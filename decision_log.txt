----------------------------------------------------------------
1. KEY ASSUMPTIONS
----------------------------------------------------------------

1.1 Data Structure
- Assumed the two CSV files (Work Orders and Deals) represent
  the core business data needed to answer founder-level queries.
- Assumed column names in Monday.com boards would remain
  consistent after CSV import, allowing reliable field mapping.
- Assumed "messy data" means inconsistent formats (e.g. "$10k",
  "10,000", missing values) rather than fundamentally broken
  data — handled via normalization in get_board_items().

1.2 Query Scope
- Assumed "founder-level business questions" means pipeline
  health, revenue metrics, sector performance, and operational
  status — not deep financial modeling.
- Assumed follow-up queries within the same session should
  retain context (e.g. "Now filter those by energy sector"
  after "Show top deals").

1.3 Live Data Requirement
- Interpreted "live API calls at query time" strictly — no
  caching, no pre-loading data at startup. Every user question
  triggers a fresh Monday.com API call.

1.4 Authentication
- Assumed evaluators should not need to provide any API keys
  or configuration. All credentials are stored server-side
  in Streamlit Cloud secrets.


----------------------------------------------------------------
2. TRADE-OFFS CHOSEN AND WHY
----------------------------------------------------------------

2.1 Cerebras over Groq / Gemini
- Tried Groq (llama-3.3-70b) first but hit strict rate limits
  causing frequent crashes during live demos.
- Tried Gemini but hit free tier quota limits quickly.
- Chose Cerebras (gpt-oss-120b) because:
    * Completely free with higher rate limits
    * Fastest inference available (sub-second responses)
    * Supports tool calling natively
    * More reliable for live evaluation demos

2.2 Streamlit over React/Next.js
- Chose Streamlit because the entire stack is Python,
  making development faster within the 6-hour constraint.
- Trade-off: Less polished UI compared to a React frontend,
  but fully functional and easy to deploy for free on
  Streamlit Cloud.

2.3 REST API over MCP
- Used Monday.com REST/GraphQL API directly instead of MCP.
- MCP would have been a bonus but adds setup complexity.
- Direct API gives full control over queries, pagination,
  and error handling without extra dependencies.

2.4 Two tools over many specialized tools
- Kept the tool set minimal: list_boards and fetch_board_data.
- Trade-off: Less granular control, but simpler for the LLM
  to reason about and less likely to cause tool-calling errors.
- The LLM handles all data analysis in its context window
  after fetching raw board data.

2.5 No pagination beyond 100 items
- Set Monday.com items_page limit to 100 per board.
- Trade-off: May miss items on large boards, but sufficient
  for the provided dataset size within the time constraint.
- Full pagination would be added with more time.


----------------------------------------------------------------
3. WHAT I'D DO DIFFERENTLY WITH MORE TIME
----------------------------------------------------------------

3.1 Full pagination
- Implement cursor-based pagination to fetch ALL items from
  large boards, not just the first 100.

3.2 Smarter data normalization
- Build a more robust parser for currency, dates, and status
  fields using regex and type inference per column.
- Currently basic — strips $ and commas, converts "10k" to
  10000, but edge cases exist.

3.3 MCP integration
- Integrate Monday.com via MCP for cleaner tool abstraction
  and bonus credit as specified in the assignment.

3.4 More granular tools
- Add specialized tools like:
    * filter_board_by_sector(sector, board_name)
    * get_deals_by_stage(stage)
    * get_overdue_work_orders()
  This would reduce token usage and improve accuracy.

3.5 Better UI
- Add charts and visualizations using Plotly or Altair
  directly in Streamlit for pipeline health, sector breakdown,
  and revenue trends.
- Add a "Export to PDF" button for leadership briefings.

3.6 Error handling
- Add more graceful error messages when Monday.com API is
  slow or returns unexpected data structures.
- Show partial results instead of failing completely.

3.7 Caching with TTL
- Add optional short-lived caching (e.g. 60 seconds) to
  reduce API calls for repeated similar queries while still
  keeping data reasonably fresh.

4. BONUS: LEADERSHIP UPDATES
----------------------------------------------------------------

Attempted: Yes (partially)

Interpretation:
- Interpreted "leadership updates" as a one-click Executive
  Briefing feature that automatically generates a structured
  weekly summary without the user needing to ask specific
  questions.

Implementation:
- Added example prompts in the UI that guide users toward
  leadership-style questions (pipeline health, sector
  performance, top deals, delayed work orders).
- The agent's system prompt is tuned to respond in an
  executive briefing style — key metric first, supporting
  breakdown second, data caveats third.

What I would build with more time:
- A dedicated "Generate Weekly Briefing" button that triggers
  a fixed prompt fetching data from both boards and producing
  a structured report with sections:
    * Pipeline Summary (total value, by stage, by sector)
    * Top 5 Deals to Watch
    * Work Order Health (on-track vs delayed)
    * Key Risks and Blockers
    * Recommended Actions
- Export this briefing as a downloadable PDF or email draft.
